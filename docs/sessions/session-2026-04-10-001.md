## Session Notes

- **Token attribution architecture:** Planning tokens → captain, task execution tokens → mission agent, verification tokens → captain. This gives accurate per-agent cost visibility in the build report.
- **Variable iteration design:** The D + S/R cycling with exit condition is a clean way to handle "2 to 5 iterations" without hardcoding. The `_run_and_record` helper prevents duplication between iteration types.
- **Two-level completion architecture:** The `_verify_mission` method is structured for future enrichment. Currently passes output file paths but not content; adding content summaries would strengthen semantic verification. The infrastructure (separate method, configurable criteria, output paths param) supports this extension.
- **Build report uses filesystem scanning** rather than tracking counters for article/word/cross-ref counts. This makes the report resilient to edge cases (pre-existing articles, agents that produce output outside the mission system).
- **FakeLLMClient in tests/helpers.py** mirrors `total_tokens_used` counter (15 tokens per call), making orchestrator budget tests work without mocking. All budget/token tests rely on this.
